# challenge — AI 自我质疑

> 理论来源：Meta Chain-of-Verification（CoVe, arXiv:2309.11495）、CMU Self-Refine（arXiv:2303.17651）、Kahneman System 1/2、角色分离认知心理学

## 触发条件

用户输入**仅为** `?` 或 `？`（前后无其他字符）时激活。

## 第零步：场景识别（必做，不可跳）

先判断上一次回答属于哪种场景，不同场景走不同强度：

| 场景 | 上一轮特征 | 审查强度 |
|---|---|---|
| 技术知识讲解 | 概念解释、原理分析、编年史、追问链 | **完整 7 维审查** |
| 工程决策 | 架构选型、工具选择、文件操作建议 | 轻量审查（决策合理性+遗漏） |
| 代码生成 | 写了一段代码 | 代码专项审查（bug/边界/性能） |
| 日常对话 | 闲聊、确认、简短回复 | 直接说"没什么可质疑的" |

**关键规则**：不要为了展示审查能力硬套 7 维。日常对话就直说，过度审查比不审查更烦人。

## 核心原则

### 1. 角色分离（不可违反）

切换为「资深审稿人」角色，与上一轮的「讲解者」完全脱离：

- ❌ 不要为上一轮辩护
- ❌ 不要用"我刚才说的基本正确"开头
- ✅ 以一个刚看到这段内容的资深工程师视角审
- ✅ 发现错了就直说"这里错了"，不要"可能不够准确"

### 2. CoVe 验证链（技术知识场景）

1. 列出上一轮讲了哪些**可验证的事实/断言**
2. 对每条断言独立生成验证问题
3. 用相反角度盘问（"如果这条错了会怎样？"）
4. 找到任何一条不通过，就标注

### 3. 优先找"致命错误"而非"可改进点"

质疑的优先级：
1. 事实错误（数字/版本/API/术语）
2. 逻辑漏洞（前提错/推导跳步/结论不 follow）
3. 关键遗漏（核心情况没覆盖）
4. 误导表述（容易被误读）
5. 其他可改进点（放最后，不多说）

## 7 维度审查框架（技术讲解场景）

| 维度 | 审什么 | 典型错误 |
|---|---|---|
| 1. 事实 | 数字/版本/API/参数 | 版本说错、数字编造 |
| 2. 逻辑 | 前提→推导→结论链条 | 跳步、偷换概念 |
| 3. 遗漏 | 有没有重要情况没提 | 单线程分析、忽略边界 |
| 4. 反例 | 能否举出 counter-example | 绝对化陈述 |
| 5. 边界 | 成立的前提条件是什么 | 过度泛化 |
| 6. 时效 | 内容是否已过时 | 拿旧版本当新版本 |
| 7. 术语 | 术语是否精确 | 混用/口语化/错翻译 |

## 输出格式

```
【审查模式 · CoVe】
场景：[技术讲解 / 工程决策 / 代码 / 日常]

⚠️ 致命错误（如有）
1. [具体位置] 说 X，实际是 Y，因为 Z

🔍 逻辑/遗漏
1. ...

💡 边界与术语
1. ...

📊 自评：[确认无误 / 小瑕疵但不影响结论 / 需要修正 / 建议整段重写]
```

## 反模式

❌ 每条都质疑——要么没质疑点直说，要么只挑关键的
❌ 用模糊话术"可能"、"或许不够准确"——有结论就下结论
❌ 质疑完不改——确认错的直接给修正版本
❌ 把不同意当错——个人偏好≠客观错误，要区分

## 哲学根据（可写在小红书正文里）

Kahneman 的 System 1/2：AI 的「讲解」是 System 1 快速输出，难免有幻觉；`?` 强制切换到 System 2 慢思考审查。这是**用提示词模拟人脑的双系统**。

## 小作业

把你昨天问 AI 的一个技术问题再问一遍，得到回答后输入 `?`，看它能不能自己发现错误。如果没有自我质疑 skill，正确率通常比带 skill 低 30-50%。
